The enforcement invariant (skill gate)

StepMechanism
1. Skill invokedSKILL.md first step: python3 .claude/scripts/raven-mark-skill.py andie-jr — marker appended with script-stamped ts
2. Model attempts Edit/WritePreToolUse fires raven-skill-gate.py with tool JSON on stdin
3. Policy checkrouting-policy.json: mode (shadow|soft|hard|off), required_skills, scope/exclude globs, freshness_hours
4. Freshnessmarker session_id matches, OR marker newer than .session-start mtime, OR within 4h fallback window
5a. Fresh marker / out of scopeexit 0 — allowed
5b. No marker, mode=hardexit 2 — BLOCKED by raven-skill-gate to stderr; model must invoke the specialist
5c. No marker, shadow/softexit 0 + violation logged to .raven/audit/ and docs/observations/security_log.md; soft also warns
Escape hatchtouch .raven/state/gate-override → next N calls allowed, every use audited — never silent

Design rule: gate the action, not the intent. No prompt classification in the hook — it enforces a workflow invariant via marker files. Honest contract: guarantees the skill ran, not that its output was used well.

Token boundary — the only three paid paths

#PathCostBound by
1Hook stdout/additionalContext injected per message~100 tok/msgkept terse by design; review bar: must earn back 60 tok × every message
2SKILL.md content when a skill loads~1–2k/sessionmode-split files — load only the selected mode, never all four
3Violation/block messages~50 eachonly on violations; zero on compliant sessions

Everything else — gate checks, scans, audits, metrics, notifications — executes in Python on the developer machine and never enters model context. Steady-state governance overhead: ~2% of session tokens.
